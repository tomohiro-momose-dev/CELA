#ルール

ルール1：議論の結論が出たら「まずドキュメントを直す」
AIとの議論で仕様の追加・変更が決まったら、コードを書く前に、まずローカルのドキュメント（docs/ 配下のファイル）を修正させます。

AIへの指示イメージ: 「議論で合意した仕様変更を、まずはローカルの docs/xxxxx.md に不足なく反映せよ」

ルール2：ドキュメント変更を「即座にGitHubへPush（またはPR）する」
ここがユーザー様の仰るコアの部分です。ドキュメントが書き換わったら、コードの実装を始める前（あるいは実装と同時）に、GitでコミットしてGitHubへ送らせます。

小さな修正なら: 直接メインブランチにPushさせてOK（特に /docs 配下はコードを壊さないので安全です）。

大きな要件変更なら: 仕様変更用のブランチを切って、GitHub上にプルリクエスト（PR）を作らせる。

ルール3：GitHubのIssue（マスター）にも「跡跡を残す」
GitHubのIssueが絶対的なマスターなので、ドキュメントを更新したコミットやPRに、必ず #1（Issue番号） を紐付けさせます。これにより、Issueのタイムラインを見れば「議論の末に仕様がどう変わったか」が自動で記録されます。




## 5. プロジェクト管理・ワークフロー規範（重要）

あなたは単なるコード生成AIではなく、当プロジェクトの自律的な開発メンバーです。以下の開発プロセスを厳守してください。

### 5-1. 仕様のマスター（絶対正義）
- 本プロジェクトの要件定義・仕様のマスターデータは「GitHub上のIssue」です。
- 作業を開始する前、および仕様に疑問が生じた場合は、必ず `gh issue view [Issue番号]` を実行し、最新のコメントまで含めて要件を再確認してください。ローカルの古い一時ファイルや自己判断を優先してはいけません。

### 5-2. 変更管理とトレーサビリティ
- ソースコードを変更した場合は、必ず対応するIssue番号（例: `#1`）をコミットメッセージまたはプルリクエストに含めてください。
- 後続フェーズに影響する制約やバグの発見など、「未来への申し送り事項」が発生した場合は、チャットでの報告と同時に、必ず `docs/README.md` の該当箇所にドキュメントとして即座に追記してください。

### 5-3. 品質ゲート
- 一つのタスク（例: Phase 1）が完了し、ビルドおよび検証が正常終了（Green）となるまでは、次のタスク（例: Phase 2）の実装を開始してはなりません。


## 4. セルフリフレクション（自己内省）プロトコル
あなたはコードを生成、またはドキュメントを書き換えた後、ユーザーに提案する前に必ず以下の「セルフチェック（内省）」を脳内で実行し、矛盾がないか確認してください。

### チェックリスト：
1. **一貫性の確認:** - 提案したコードは、`docs/` 配下の最新の設計書（例: コマンドがスペース区切りであることなど）と100%矛盾していませんか？
2. **境界値・安全性の確認:**
   - C++コードにおいて、バッファオーバーフローのリスクはありませんか？（例: 12桁の文字列に対して、ヌル文字を含めた13バイトが正しく確保されているか）
3. **Mbed OS 6 / C++11の適合性:**
   - 使用している関数や文法は、ターゲット環境で非推奨になっていませんか？

### 行動規範：
もし上記チェックの段階で少しでも「矛盾」や「潜在バグ」を見つけた場合は、ユーザーに隠さず、「内省の結果、〇〇の矛盾に気づいたため、修正しました」と報告した上で、修正済みのクオリティの高いコードを提案してください。

## 6. 仕様変更時のGitHub同期プロトコル（重要）

AI（あなた）とユーザーとの議論の結果、要件定義や仕様の修正・追加が発生した場合、以下の手順で即座にGitHub（マスター）と同期してください。

### 同期の手順：
1. **ドキュメントの更新:** 議論の結論を、まずローカルの `docs/` 配下の該当Markdownファイルに正確に反映してください。
2. **Gitコミットの作成:** 変更したドキュメントをステージし、メッセージに変更内容と関連Issue番号（例: `docs: 議論に基づきMACアドレスのバリデーション仕様を追加 #1`）を含めてコミットしてください。
3. **GitHubへの即時反映:** - 軽微なドキュメント修正の場合：そのまま `git push` を実行し、GitHubのマスターを最新に更新してください。
   - 大幅な仕様変更の場合：新ブランチを作成してPushし、`gh pr create` で仕様変更のプルリクエストを作成してユーザーの承認を得てください。
   
ユーザーから「コードの実装に進んで」と言われても、このドキュメントのGitHub同期が完了するまでは、次のコード変更を行ってはなりません。

2. OpenRouter 等の外部監査（強い分離）

生成物（diff / ファイル） + 要件 + （任意）チャット要約
        → OpenRouter 上の別モデル

呼び方の例:

Cursor から Shell で監査スクリプト実行（Skill に「実装後はこれを走れ」と書く）
Cursor SDK / CI で Agent.prompt(..., model=監査用) の二段パイプライン
Hook（stop / after edit）でスクリプト起動 ※環境次第
強み: モデル・プロンプトを完全分離、決め打ち検知に向く
弱み: API キー・スクリプト・コンテキスト受け渡しの設計が必要

3. Rules だけの「自己内省」（弱いがゼロコスト）
alwaysApply や Skill で:

計画・実装の直後に、変更理由が要件にない定数/API/バッファ変更がないか自己監査し、違反は修正前に列挙せよ

これは 同一系統のモデルの自己チェック なので、あなたの求める「別モデル監査」の代替にはなりません。補助にはなります。
