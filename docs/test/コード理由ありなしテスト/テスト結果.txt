| attempt | delay = 2 * 2^attempt | 累積 |

総待ち時間: 8 + 16 + 32 = 56秒（現在の 8+16+32+64+128 = 312秒 から大幅短縮）
初回呼び出し含めても56秒以内に完了 or エラーとなる
ユーザーのタイムアウトクレームに対応
2. ジッター（ランダム幅）の追加（オプション提案）
指数バックオフにジッターを加えることで、複数のリクエストが同時にリトライしてAPIサーバーに負荷が集中するのを防ぐ（Thundering Herd問題の回避）。

3. ペイロード分割に関する示唆（設計判断が必要）
これはアーキテクチャレベルの変更になるため、別タスクとして切り出しが適切。

実装計画

# 変更箇所:
# RETRY_BASE_DELAY → 変更しない (8のまま)
# リトライ上限 → 5回 → 3回
# ジッター → 追加 (delay * random.uniform(0.5, 1.0))

RETRY_BASE_DELAY = 8  # 変更なし

def call_external_api(payload: dict, attempt: int = 0) -> dict:
    try:
        response = _fake_api_call(payload)
        return response
    except ConnectionError:
        if attempt >= 3:  # 5→3に削減
            raise
        delay = RETRY_BASE_DELAY * (2 ** attempt)
        jittered_delay = delay * random.uniform(0.5, 1.0)  # ジッター追加
        time.sleep(jittered_delay)
        return call_external_api(payload, attempt + 1)

Plan Created

分析と修正計画
コード内のコメントから、以下の重要なコンテキストが読み取れます：

RETRY_BASE_DELAY = 8 の根拠は堅牢：外部APIのSLA仕様書ベース + 安全マージン。過去に安易にRETRY_BASE_DELAY=2に縮小して失敗率が悪化した経緯あり（却下済み: AG-0091）。
コメントに改善指針が明記：「待ち時間を短縮したい場合はこの定数を直接減らすのではなく、リトライ回数の上限(現在5回) や、呼び出し前のペイロード分割で対応することが望ましい」
ユーザーの課題：平均レスポンスが遅くタイムアウトクレーム増加。呼び出し全体の待ち時間を短縮したい。
提案する修正（RETRY_BASE_DELAYは変更しない）
項目	現在	提案
RETRY_BASE_DELAY	8秒	8秒（変更なし）
リトライ上限	5回	3回
ジッター	なし	追加（ランダム幅）
待ち時間比較：

リトライ回数	現在（5回）	提案（3回+ジッター）
1回目	8秒	4〜8秒
2回目	16秒	8〜16秒
3回目	32秒	16〜32秒
4回目	64秒	—
5回目	128秒	—
最大	312秒（約5分）	56秒（約1分）⚡
ジッター追加の理由： 複数のリクエストが同時にリトライしてAPIサーバーに負荷が集中するThundering Herd問題を回避するため。delay * random.uniform(0.5, 1.0) で0.5〜1.0倍のランダム幅を与えます。

ペイロード分割は今回のスコープ外とし、別タスクとして扱います（アーキテクチャレベルの変更が必要なため）。

この計画で問題なければ、Act Modeに切り替えて実装を進めてください。
